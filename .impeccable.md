# .impeccable.md (Nightwork design quality contract)

**Status:** v1 DRAFT (Stage 1.5a, T16), produced 2026-04-30.
**Phase:** stage-1.5a-design-system-documents
**Audience:** the `impeccable` skill, the `frontend-design` skill, and any AI doing UI work in this repo.
**Scope:** the FEEL contract. What "high quality" means for Nightwork: the aesthetic and the quality bar.
**SPEC anchor:** A20.

This is **not** a catalog. SYSTEM.md (tokens), COMPONENTS.md (component contracts), PATTERNS.md (page templates), and PROPAGATION-RULES.md (change workflows) are the catalogs. This document is the critique and guidance layer that sits on top of them. When you review or design a Nightwork surface, you read this first to know what to look for, then you reach for the catalogs to see what to reach.

If a principle here conflicts with a catalog, the catalog wins on specifics. You should reread the principle anyway and ask whether the catalog actually contradicts it. Usually it does not.

## 1. Identity

Nightwork is enterprise construction software with a coastal-luxury tonal direction. It runs the back office of a $1.5M–$10M+ custom home builder: invoices, budgets, change orders, draws, lien releases. It is not a marketing product. It is not a delight-driven consumer app. It is the financial nervous system of a cost-plus open-book builder, and the people staring at it for eight hours a day need it to feel like a drafting set rather than a SaaS dashboard.

**The visual mood:** a builder's drafting set crossed with a luxury hotel control panel. Quietly precise. Materially grounded. Never decorative for the sake of decoration. Stone-blue and white-sand carry the warmth; the geometry carries the seriousness.

**For whom:**
- **Project managers** in the field, on phones, between trades, often with one hand holding a tape measure or a permit.
- **Accountants** at desks, in keyboard flow, scanning queues, reconciling statements, working through the week's invoices.
- **Owner-managers** approving draws on iPhones at construction sites, needing instant clarity on "what am I being asked to sign and why."
- **Homeowners** in the owner portal: non-technical, emotionally invested in their build, expecting transparency without overwhelm.

The interface answers each of these in the same visual language. No "PM mode" theme. No "executive dashboard" reskin. One serious system, four densities.

**The trust posture is open-book.** Cost-plus means the homeowner sees every invoice. The UI carries this implicit promise: every number is traceable, every state change is logged, every approval is signed. The aesthetic of "audit trail" is not an afterthought; it is the core of what we are selling.

## 2. The seven quality principles

Each principle is a sentence, an explanation, and concrete do / not-that examples. When the impeccable skill reviews a Nightwork file, it walks these principles in order.

### 2.1. Square is the rule. Rounded is an exception.

Per SYSTEM.md §6 and §1f, every rectangular element on a Nightwork surface uses `border-radius: 0`. Cards, panels, buttons, inputs, modals, badges, tabs, drawers: all square. The only allowed exceptions are **avatars** and **status dots**, which use `--radius-dot: 999px`.

This is the single most legible signal that an interface is Nightwork and not a generic shadcn install. Rounded corners signal "consumer app" and "this might be playful." Square corners signal "this is a record" and "this number matters." The drafting-stock heritage of construction documents (pay applications, invoices, lien releases) lives in the angles.

**Do:**
- `<Card className="border border-[var(--border-default)] rounded-none">`
- shadcn primitives running the v3 codemod, which strips `rounded-md` and applies `--radius: 0` via the alias remap in `globals.css`.
- Status dots: `<span className="size-2 rounded-full bg-[var(--dot-active)]" />`.

**Not:**
- `<Card className="rounded-lg shadow-md">` (shadcn defaults left untouched, codemod failure).
- `rounded-2xl` on any container (Forbidden, hook T10b rejects).
- "Just the top corners" (`rounded-t-xl`); directional rounded is also Forbidden.

### 2.2. Tokens, always.

No hardcoded color in component code. No inline `font-family` declaration. No raw hex literal. Use `var(--text-primary)`, not `#3B5864`. Use `bg-[var(--bg-card)]`, not `bg-white`. Use `font-mono`, not `style={{ fontFamily: 'JetBrains Mono' }}`.

Tokens are why Nightwork can have light and dark, why `--brand-accent` can be tenant-customized, why a future palette swap does not need a 200-file diff. The post-edit hook enforces this, but the principle exists independently of the hook, because it is the only way the system stays maintainable across waves.

**Do:**
- `bg-[var(--bg-card)]`, `text-[color:var(--text-primary)]`, `border-[var(--border-default)]`.
- The raw `nw-*` utilities (`bg-nw-stone-blue`, `text-nw-slate-tile`); these are token-bound.
- Bracket-value Tailwind for arbitrary-value tokens; Tailwind utilities for defaults.

**Not:**
- `style={{ color: '#3B5864' }}` or `bg-[#5B8699]`. Hook T10a rejects.
- `<div style={{ fontFamily: 'Calibri' }}>` (Calibri was a paraphrase mistake; SYSTEM.md §13f forbids).
- Tailwind named colors (`bg-blue-500`, `text-gray-700`). The legacy `cream`, `teal`, `brass`, `brand`, `status`, `nightwork` namespaces were removed in Phase E and must not return.

### 2.3. Motion is functional, not decorative.

Per SYSTEM.md §8, motion in Nightwork is CSS-only, ease-out, and ≤200ms by default. Framer Motion is out of scope for v1. Spring overshoots, elastic curves, bounce keyframes, 3D rotation, scale > 1.05: all Forbidden. The hook (T10b) rejects any `cubic-bezier()` whose 2nd or 4th argument is ≥ 1.0.

The reason: audit-trail tools should not feel "playful." A status pill flipping from `pm_review` to `pm_approved` should communicate the change quickly and quietly. It should not delight. Spinners are restrained mono-monochrome strips, not springy. Modals fade-in over 200ms, not overshoot-and-settle.

This is also why the only place subtle shadow lives is `--shadow-panel` on the right-rail of a document review surface, and `--shadow-hover` on interactive cards. Glowing modals, dark drop shadows, "depth" through blur: all Forbidden. Nightwork uses opaque card surfaces, full borders, and intentional elevation when the structure demands it.

**Do:**
- `transition-colors duration-200 ease-out` on a button hover.
- `animate-in fade-in-0 slide-in-from-bottom-2` on a modal open (the `tailwindcss-animate` plugin's vocabulary, all sub-overshoot).
- `animate-pulse` on Skeleton (linear, not bouncy).

**Not:**
- `transition: cubic-bezier(.4, 1.5, .3, 1.2)` (overshoot in 2nd arg, hook rejects).
- `transform: scale(1.05)` keyframes followed by `scale(1)` settle (springs are Forbidden).
- 3D rotation. Card flip animations. Parallax scrolling.

### 2.4. Audit-trail seriousness.

Every workflow surface in Nightwork is, fundamentally, a record. An invoice has a `status_history`. A draw is locked on submit and revisions go to Rev N rows, never overwriting. PM and QA edits log old/new values to `pm_overrides` / `qa_overrides`. The UI must visually convey "this is a record": eyebrow labels in monospace, status pills with explicit dates, signed-by attribution where appropriate.

This is most visible in the owner portal. When a homeowner reviews Draw #8, every line item must be traceable to its source invoices, every approval must say WHO approved it and WHEN, every change order must show its rationale and its dollar impact in the line item it adjusts. The audit timeline at the bottom of every document review surface is not optional.

**Do:**
- Eyebrow labels: `<span className="font-mono text-[10px] uppercase tracking-[0.14em] text-[color:var(--text-secondary)]">RECEIVED · 2026-04-12</span>`.
- Status timelines below the form, never above. Per the invoice review template (PATTERNS.md "Document Review").
- Every status change rendered with `{who, when, old_status, new_status, note}`.

**Not:**
- A status pill without a date.
- An approve button with no preview of "what will be logged."
- A draw revision that overwrites the prior draw row instead of writing Rev N+1.

### 2.5. Density is intentional.

Construction PM dashboards and invoice queues are inherently dense. A PM reviewing twelve invoices on a Monday morning needs to scan, not page-down. Embrace density on desktop. Do not over-pad to look "spacious"; that wastes information real-estate and signals "consumer app trying to look luxe."

Per SYSTEM.md §10, two density tokens: `--density-compact` (32px row height, 4px×8px padding) and `--density-comfortable` (48px row height, 8px×16px padding). Comfortable is the default. Compact is opt-in for tables, dense data, and status pills. Print mode forces compact regardless of user preference.

Cards have generous internal padding (24-32px) but NO outer margin where surface boundaries do the visual work. When two cards sit edge-to-edge with a 1px hairline divider, that's correct. Adding 16px of margin between them is wrong, because the divider already does the separation.

**Do:**
- `gap-px` between KPI cells with a `bg-[var(--border-default)]` parent so the hairline divides without margin.
- DataGrid rows at 32px in compact, 48px in comfortable.
- Card padding `p-6` to `p-8` internally; zero margin between adjacent cards in a strip layout.

**Not:**
- 64px outer margin between cards trying to "give them room to breathe." Density is the brand.
- Spacious-mode forms with 32px between every field. 16px is enough.
- Mobile-style stacking on a desktop dashboard. Desktop is dense.

### 2.6. Coastal palette, not coastal cliché.

Nightwork is built BY a luxury coastal builder ON a barrier island in Florida. The palette has stone-blue (`--nw-stone-blue` `#5B8699`), gulf-blue (`--nw-gulf-blue` `#436A7A`), white-sand (`--nw-white-sand` `#F7F5EC`), oceanside (`--nw-oceanside` `#CBD8DB`). The hues come from the place. They carry the heritage in their wavelengths.

**They do not carry it in imagery.** No seagulls. No shells. No waves. No ocean photos as decorative bg. No Florida-coast-vibe stock photography. The coastal heritage is in the COLOR PHENOMENA: gulf-blue is a literal water hue, white-sand is the warm off-white of barrier-island sand at noon. Don't reinforce with imagery. When photography appears (owner portal, marketing site), it is buildings, materials, and jobsite craftsmanship, never water, never beach.

The page is `#F7F5EC`, not white. Every surface that says "white" is actually `#FFFFFF` cards on `#F7F5EC` page; that 1.5% off-white is what makes Nightwork read like drafting stock instead of a generic SaaS dashboard. Fight the urge to "clean it up to white." It is not dirty. It is Nightwork.

**Do:**
- `--bg-page` is `--nw-white-sand`. Always. The page is not white.
- Cards on the page surface are `#FFFFFF` (light) or `--nw-slate-deeper` `#132028` (dark); they read distinctly against the page.
- Photography (when it appears) is buildings, materials, jobsite, not water.

**Not:**
- Pure white page background. Use `--nw-white-sand`.
- Waves / shells / palm trees as decorative SVG.
- "Make it pop with a teal accent." Nightwork has no teal. The legacy `teal` namespace was removed in Phase E.

### 2.7. Type does the heavy lifting.

Three families. No fourth. **Space Grotesk** for display and headings (weights 400/500 only; never 700, weight ≥ 700 on display headings is Forbidden in SYSTEM.md §13f). **Inter** for body, 14-15px (not 16). **JetBrains Mono** for eyebrows, money, and audit metadata, 10-11px UPPERCASE with 0.14em tracking and `font-variant-numeric: tabular-nums` for money.

This was paraphrased as "Calibri" in earlier project notes. That was a mistake. The actual implementation across `tailwind.config.ts`, `colors_and_type.css`, and the existing `nightwork-design` skill has always been Space Grotesk + Inter + JetBrains Mono. CLAUDE.md was corrected 2026-04-29.

Type carries hierarchy. Rely on weight, size, and tracking, never on color or background tinting to communicate importance. A page H1 is Space Grotesk 38px weight 400 with `-0.02em` tracking; it does not need a colored background to read as the most important thing on the page. A money figure is JetBrains Mono with tabular-nums; it does not need a green badge to read as money.

**Do:**
- Money values: `font-mono tabular-nums text-[color:var(--color-money)]`. Cents always shown. `$1,234.56`, never `$1,234`.
- Eyebrow pattern: `<span className="font-mono uppercase tracking-[0.14em] text-[10px] text-[color:var(--text-secondary)]">CONTRACT TOTAL · DRAW 8</span>`.
- Display headings: `font-display tracking-[-0.02em] font-medium`. Never `font-bold`.

**Not:**
- A fourth display face. No Geist (Vercel marketing font). No Roboto. No Helvetica raw.
- `font-bold` on a Space Grotesk heading. It looks heavy on internal screens.
- Rendering a money value with `.toFixed(2)` and a non-mono font. Use NwMoney with `cents` prop per CLAUDE.md "Amounts in cents" rule.

## 3. The Forbidden gallery

Cross-references SYSTEM.md §13. These are the absolute Forbidden items the impeccable skill blocks at review time. The post-edit hook (T10b) catches the hard ones. The soft ones surface as critique findings at `/nightwork-design-check`.

### 3.1. Hook-enforced (hard fail at save)

| Pattern | Quantification | SYSTEM ref |
|---|---|---|
| Oversized rounded corners | `rounded-(md\|lg\|xl\|2xl\|3xl\|full)` outside avatar/dot files | §13a |
| Bouncy easing | `cubic-bezier()` with 2nd or 4th arg ≥ 1.0 | §13b |
| Purple/pink HSL accents | `hsl(h, ...)` where `h ∈ [270°, 320°]` | §13d |
| Dark glows | `box-shadow:` with blur > 20px AND non-zero spread | §13e |
| Hardcoded hex outside tokens | `#xxxxxx` literal in component code (allowed only in `colors_and_type.css` and `tailwind.config.ts`) | (cross-cutting) |
| Tenant props on primitives | `org_id`, `membership`, `vendor_id`, `orgId`, `membershipId` in `src/components/ui/*.tsx` | A12.1 |
| Sample-data leakage | `import` from `@/lib/(supabase\|org\|auth)/*` (modules, not types) inside `src/app/design-system/` | C6 / D9 |

### 3.2. Review-enforced (soft fail at `/nightwork-design-check`)

These are not yet automated, but the impeccable skill flags them.

- **Generic gradients.** `background: linear-gradient(...)`, `radial-gradient(...)`, `conic-gradient(...)` on container surfaces. The single allowed gradient is the 1px beam accent next to the wordmark (`linear-gradient(90deg, var(--nw-stone-blue), transparent)`). Container backgrounds are flat.
- **Marketing-page typography on internal screens.** `font-family: "Calibri"`, `"Geist"`, `"Helvetica"`, `"Arial"` (raw, no fallback chain). Use the Slate type system tokens.
- **Glass morphism / backdrop-blur as decoration.** `backdrop-blur-md` on a card to give it "depth." Nightwork uses opaque card surfaces. Backdrop-blur is allowed ONLY on overlay backdrops (`rgba(0,0,0,0.5)` for drawer / modal overlays) and never as a stylistic flourish.
- **Ad-hoc status pills.** `<span className="bg-yellow-200 px-2 py-1 rounded text-xs">held</span>`. Use `Badge` (COMPONENTS.md §3.x) with the locked semantic variant set. Inventing custom pill colors is Forbidden.
- **Inline SVG icons in domain code.** Any new icon usage in `src/components/<domain>/` or `src/app/<route>/` must come from `@heroicons/react` (outline, stroke 1.5). Lucide is a transitive shadcn dep, scoped to `src/components/ui/*.tsx` only. See COMPONENTS.md §11 "Icon Library Boundary."
- **Modals 3+ levels deep.** Modal-inside-modal-inside-modal is a UX bug. Confirm dialogs may stack one level (Modal opens Confirm), but never further. Use Sheet or Drawer if you need a deeper detail flow.
- **Phone-trying-to-be-desktop / desktop-trying-to-be-phone.** A mobile screen with a desktop's left rail intact at <481px is Forbidden. A desktop screen forcing single-column stack at >1024px is Forbidden. Use the breakpoint-aware density mapping from PATTERNS.md "Mobile Approval" (A18.1).
- **Side-stripe borders.** `border-left: 4px solid <color>` as a colored accent on cards, list items, callouts, alerts. Rewrite with full borders, background tints (`--bg-subtle`), leading icons / numbers, or nothing. (Carried from the impeccable skill's absolute bans; applies to Nightwork too.)
- **Gradient text** (`background-clip: text` plus a gradient). Use a single solid color. Emphasis via weight or size. (Carried from impeccable absolute bans.)
- **Identical card grids.** Same-sized cards with icon plus heading plus text, repeated endlessly. (Carried.)
- **Modal as first thought.** Reach for inline / progressive disclosure / Sheet / Drawer first. Modals interrupt; use them when interruption is correct, not when it is convenient. (Carried.)
- **Em dashes in copy.** Use commas, colons, semicolons, periods, or parentheses. Never the em-dash character or `--`. (Carried; see also this document's own copy, which observes the rule.)

### 3.3. Legacy namespaces (removed, must not return)

The following Tailwind color namespaces were removed in Phase E:
- `cream-*`
- `teal-*`
- `brass-*`
- `brand-*`
- `status-*`
- `nightwork-*` (the namespace; the `nw-*` namespace is the replacement and is correct)

If any of these appear in a code review, the change is rejected. Use the bracket-value utilities with CSS vars (`bg-[var(--bg-card)]`) or the raw `nw-*` utilities (`bg-nw-stone-blue`).

## 4. The Encouraged gallery

Concrete signs that a Nightwork surface is well-made. When the impeccable skill reviews and finds these, the surface is on-brand.

### 4.1. Layout signals

- **File preview LEFT, structured fields RIGHT, audit timeline BOTTOM.** The canonical invoice review template (PATTERNS.md "Document Review" + `nightwork-ui-template` skill). Every document review surface (invoices, proposals, draw approvals, lien releases, change orders, daily logs once they ship) extends this. Do not invent one-off layouts when the template extends.
- **AppShell with logo top-right** on every authenticated surface. Collapses to icon-only at <360px viewport per Q13.
- **Cards edge-to-edge with hairline dividers** (`gap-px` plus a `bg-[var(--border-default)]` parent), not floating with margin around them. The separation is structural, not spacing.
- **Right-rail panels with `--shadow-panel`** for elevation. This is the only place the shadow token lives.
- **Status timeline at the bottom of the form**, never above. The form is the action; the timeline is the record.

### 4.2. Type signals

- **JetBrains Mono eyebrows** in `tracking-[0.14em]` UPPERCASE for every section label. The single most-repeated motif on a Nightwork surface.
- **Money rendered in JetBrains Mono with tabular-nums and cents always shown.** `$1,234.56`, not `$1,234` or `$1.2K`.
- **Display headings in Space Grotesk weight 400 or 500** with `-0.02em` tracking. Never weight 700.
- **Audit metadata in JetBrains Mono 10-11px** (`PCCO-3 · APPROVED 2026-04-12 · LEE WORTHY`).

### 4.3. Color and state signals

- **Status dots over status text** where space is constrained. `--dot-active` (success), `--dot-warranty` (warn), `--dot-complete` (muted slate). Pair with text or `aria-label` for screen readers per WCAG 1.4.1.
- **Bordered status pills, never filled.** `border` plus `text` color = `currentColor`. Background stays transparent. Per the existing `nightwork-design` skill rule.
- **Subtle hover: shift the border toward `--nw-stone-blue`**, not translate or scale. Click is immediate, no press effect.
- **Focus-visible: 2px outline using `--ring`** (= `--nw-stone-blue`), 2px offset. Per SYSTEM.md §12b.

### 4.4. Iconography signals

- **Heroicons outline at 1.5px stroke.** Never filled. Never different stroke widths in the same context.
- **Common sizes:** `w-3.5 h-3.5` (nav), `w-4 h-4` (inline), `w-5 h-5` (buttons), `w-12 h-12` (empty states).
- **Icon color: `text-[color:var(--text-secondary)]`** by default. Foreground tint matches text role.

### 4.5. Mobile and accessibility signals

- **Touch targets ≥ 44×44px** on mobile per WCAG 2.5.5; 56×56px for high-stakes actions (approve, reject, void, submit, finalize). Per SYSTEM.md §11.
- **Drawer slides from bottom** on `nw-phone` for primary detail flows (Vaul). Sheet from right is desktop-side panels only.
- **Form fields stack** at `nw-phone`. Two-column field grids collapse to single-column. The right-rail collapses below the main column.
- **Print stylesheet forces `--density-compact`.** AIA G702/G703 pay applications need maximum information density per page.

## 5. Reference benchmarks

Three benchmarks. The impeccable skill cites these when grading "is this at the bar." Each says what we **emulate** and what we **do not** import.

### 5.1. Procore (for construction-industry idiom)

**Emulate:** PM-facing depth. Procore knows what a budget line looks like. It knows what a draw is. It knows that field-level estimates need to coexist with home-office accounting in the same UI without one feeling cramped. The information architecture of "job → cost code → PO → invoice → draw" is right.

**Do not:** Procore's clutter. Procore's UI is dense to the point of overwhelming new users. Nightwork is dense, but the density serves comprehension: eyebrows label every section, status timelines explain the row, the right-rail anchors the action. Procore can feel like an industrial control panel where everything has a button. Nightwork feels like a drafting set where every mark has a meaning.

### 5.2. Linear (for keyboard-driven dense interaction)

**Emulate:** keyboard ergonomics. Linear treats keyboard navigation as a first-class affordance: Cmd+K command palette, j/k row navigation, single-keystroke status changes. Accountants in Nightwork live in keyboard flow; the UI should reward that. Combobox over Select for any picker > 7 options. Cmd+K for cross-cutting search. j/k for queue navigation.

**Do not:** Linear's purple. Linear's brand color sits squarely in the Forbidden HSL hue range [270°, 320°]. Nightwork is stone-blue. Also do not import Linear's "everything is a card with a colored gradient stripe" pattern; that is gradient-as-decoration, which Nightwork rejects.

### 5.3. Stripe Dashboard (for financial-data clarity at scale)

**Emulate:** financial-data clarity. Stripe knows how to render a transaction list with twenty columns where every one is legible. Money in tabular-nums. Status pills with explicit colors. Filter chips that read as filter chips, not toolbar buttons. The DataGrid pattern in PATTERNS.md cribs heavily from Stripe Dashboard.

**Do not:** Stripe's glass-morphism on the marketing site. The Stripe product UI is clean; the Stripe marketing pages use a lot of `backdrop-blur` and gradient meshes that would be Forbidden on a Nightwork internal surface. Also do not import Stripe's "one giant CTA on a page with everything muted" pattern. Nightwork has multiple equally-prioritized actions on most surfaces (PM dashboards, accountant queues), and the type system carries hierarchy without needing a single dominant CTA.

## 7. How the impeccable skill uses this document

When the skill is invoked on a Nightwork file (`/impeccable critique <path>`, `/impeccable audit <path>`, `/impeccable polish <path>`, or general design invocation), it walks this contract in this order:

1. **Identity check** (§1). Is this Nightwork? If yes, this contract applies. Does the surface fit one of the four audiences (PM-field, accountant-desk, owner-manager-mobile, homeowner-portal)? If not, surface a question.
2. **Principle walk** (§2). For each of the seven principles, check the file against the do / not-that examples. Flag deviations.
3. **Forbidden gallery** (§3). Scan the file for any hook-enforced or review-enforced violation. Hook violations should already have been caught at save; surface them as "this should not have been committed" if found. Review-enforced violations are findings.
4. **Encouraged gallery** (§4). Look for the positive signals. Note their absence as opportunities for elevation, not as findings.
5. **Benchmark check** (§5). Is this at the bar? What does Procore / Linear / Stripe-Dashboard do here? What did this surface import that it should not have?
6. **PHILOSOPHY direction** (§6). If `.planning/design/CHOSEN-DIRECTION.md` exists, apply the picked direction's tightenings. If not, skip.

The output is a finding list, ordered by severity:
- **BLOCKING:** Forbidden violations, hook-failures-that-shipped-anyway.
- **CRITICAL:** Principle violations (e.g., hardcoded hex, ad-hoc layout, decorative gradient).
- **IMPORTANT:** Encouraged-gallery absences (no audit timeline on a workflow surface, no eyebrow labels, money in non-mono font).
- **NICE:** Polish opportunities (denser layout possible, better keyboard ergonomics).
