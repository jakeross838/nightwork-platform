## Test methodology

The hook reads from stdin a JSON object describing the tool invocation (file_path),
then reads the file from disk and runs grep checks against it. A rejection is
signaled by:

  - exit code 2
  - JSON output `{ "decision": "block", "reason": "[nightwork-post-edit] …" }`

Each test below:

1. Stages a violating change in a temp file under the repo
2. Invokes the hook with that file_path
3. Asserts exit code == 2 AND the rejection reason names the expected violation category
4. Removes the temp file

For testing locally / by hand, the manual command sequence (run from repo root):

```bash
echo '{"tool_input":{"file_path":"<absolute-path-to-temp-file>"}}' | \
  bash .claude/hooks/nightwork-post-edit.sh
echo "exit code: $?"
```

Exit code 2 = rejection (expected for these tests).
Exit code 0 = passed (would mean the hook FAILED to catch the violation — test fail).

## Test 2 — Forbidden bouncy easing hook (T10b)

**SPEC criterion:** A2.1 / D5.1 — `cubic-bezier` with 4th arg ≥ 1.0 (overshoot)
or 2nd arg ≥ 1.0 (early-spring) is forbidden.

**Setup:** Created `__tests__/_test-bouncy.css` (path: outside `/design-system/`,
so the isolation hook doesn't fire) with:
`.bouncy { transition: transform 200ms cubic-bezier(0.4, 1.5, 0.3, 1.2); }`

The 2nd arg `1.5` (early-spring) and 4th arg `1.2` (overshoot) BOTH violate.

**Result:**

```
[forbidden-A2.1] Bouncy easing — cubic-bezier 4th arg >= 1.0
(overshoot/elastic forbidden per SPEC A2.1):
1:.bouncy { transition: transform 200ms cubic-bezier(0.4, 1.5, 0.3, 1.2); }

[forbidden-A2.1] Bouncy easing — cubic-bezier 2nd arg >= 1.0
(forbidden per SPEC A2.1):
1:.bouncy { transition: transform 200ms cubic-bezier(0.4, 1.5, 0.3, 1.2); }
```

Exit code: 2. **PASS** — both checks fired.

Path note: the hook's file scope filter (`src/* | */src/* | supabase/migrations/*`)
deliberately excludes `__tests__/*`, so to get the hook to fire on a test, we
created the temp file at `src/__tests__/_test-bouncy.css` instead. The hook
correctly recognized the path, ran the .css block, and rejected.

## Implementation note — POSIX vs PCRE regex

The first attempt at the T10c sample-data isolation hook used a negative
lookahead `(?!types['"]|types/)` to allow type-only imports. `grep -P`
(PCRE) is unavailable in some Windows Git Bash locales (returned
"supports only unibyte and UTF-8 locales" error). The shipped hook uses
a two-pass approach:

  Pass 1: grep -nE matches every `from '@/lib/(supabase|org|auth)…'` import
  Pass 2: awk filters out the allowed `@/lib/supabase/types`-rooted lines
  Anything left after pass 2 is forbidden and rejected.

This is functionally equivalent to the lookahead form and is cross-platform.
