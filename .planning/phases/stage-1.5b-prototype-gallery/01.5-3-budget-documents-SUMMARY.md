## What was built

### Task 1: Budget view prototype

`src/app/design-system/prototypes/jobs/[id]/budget/page.tsx` (~485 LoC).

Render shape:
1. **Header band** — JetBrains Mono breadcrumb (`Prototypes › j-caldwell-1 › Budget`) + Space Grotesk H1 (`Caldwell - 712 Pine Ave — Budget`) + sub `30 budget lines · Current draw: Pay App #5`.
2. **KPI strip** — 4-cell `grid grid-cols-2 md:grid-cols-4` with 1px gaps drawing var(--border-default) divider lines. Cells: Original contract / Revised total / Invoiced to date / Remaining. All values `<Money cents={...} size="xl" variant="emphasized" />` with JetBrains Mono accent sub-text.
3. **DataGrid** — TanStack Table v8 with 8 sortable columns (cost code, original, revised, previous, this period, total to date, % complete, balance) over 30 Caldwell budget lines. Compact 7-row-height rows + 11px font + JetBrains Mono tabular-nums. `% complete` cell renders a horizontal progress bar (40-80px wide) tinted by Site Office accent: Stone Blue for 0-99%, success-green at 100%, warning when over-budget. `balance` cell flips to `--nw-danger` when negative. Sort indicators use Heroicons `ChevronUp` / `ChevronDown` / `ChevronUpDown` (Lucide-free per A12.2 boundary).

R.2 compliance:
- `CaldwellBudgetLine` type only has `{ id, job_id, cost_code_id, original_estimate, revised_estimate }`. Not `previous_applications` etc.
- `BudgetRow` (component-local type) extends with the 5 computed fields.
- `useMemo([job, currentDraw.id])` builds rows: filter `CALDWELL_INVOICES` by `cost_code_id` and bucket on `draw_id` (excluding null + current = `previous_applications`; equal-to current = `this_period`; null is excluded from both per the canonical "drawn vs not drawn" rule).
- `useMemo([rows, job])` builds `kpis`: 4 sums + approved-CO total + over-budget line count.

Hooks order (Rules of Hooks):
- L75-79: `currentDraw` resolution (plain const).
- L83: `rows` useMemo.
- L122: `kpis` useMemo.
- L143: `sorting` useState.
- L145: `columns` useMemo.
- L255: `useReactTable` (counts as a hook because it sets internal state).
- L267: early return guarded on `!job || !kpis`.

Per CONTEXT D-22 hardening (planner C1+C2 fix): all hooks above the guard; guard precedes JSX render only.

## Tech debt for 1.5b-followup

These do NOT block the plan but were surfaced during execution. Add to `.planning/followups/1.5b-followup.md` if it doesn't already track them:

1. **Lien-release fixture covers only 2 of 4 Florida statute types.** Wave 0 SUMMARY (line 69) explicitly accepted this gap because the 11/2025 Drummond window contained no `_final` releases (a substantially-complete sub release hasn't happened yet). The component side renders all 4 correctly the moment fixture data exists. Owner: Wave 0 / fixture maintenance — re-run `scripts/sanitize-drummond.ts` once a `_final` release lands in source data, OR synthesize one for prototype completeness (parallel to the T&M synth at nwrp38).

2. **DataGrid sticky header.** The current implementation does not sticky the `<thead>` when the user scrolls within the table. At 30 rows × 28px row-height ≈ 840px tall, the header scrolls out of view on a standard desktop (~720px usable area below the KPI strip). Add `position: sticky; top: 0;` to the header row's `<tr>` in a polish pass — needs a wrapping container with explicit `max-height` and `overflow-y-auto` to anchor the sticky context. Out-of-scope per the plan's 8 verbatim line items but a UX rough edge.

3. **% complete progress bar accessibility.** The bar is currently `aria-hidden="true"` with the % textual readout adjacent. ARIA-progressbar markup (role="progressbar" + aria-valuemin/now/max) would be more announceable. Polish pass — every component playground rule says ARIA semantics are non-negotiable for interactive UI; a visual readout is borderline.

4. **DataGrid filtering.** The plan delivered sortable columns but no filter input — components/data-display playground has a global filter input (line 466-484). Adding it here would make the 30-row grid more navigable. Out-of-scope per plan, polish for later.
